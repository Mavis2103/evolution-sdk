# Enterprise Addresses

Enterprise addresses contain only a payment credential, with no staking component. They're designed for scenarios where staking is not needed or desired.

## Structure

```
Enterprise Address = Payment Credential Only
```

**Payment Credential**: Controls who can spend UTXOs at this address  
**No Staking**: Cannot delegate stake or earn staking rewards

## When to Use

- **Exchanges**: Custodial platforms managing large volumes
- **Smart contracts**: Script addresses that don't need staking
- **Hot services**: Applications prioritizing simplicity over rewards
- **Testing**: Quick address generation without stake key setup
- **Privacy**: Avoiding stake key linkability across addresses

## Manual Construction

Create enterprise addresses by instantiating the `EnterpriseAddress` class:

```typescript twoslash
import { EnterpriseAddress, KeyHash, Address } from "@evolution-sdk/evolution";

const address = new EnterpriseAddress.EnterpriseAddress({
  networkId: 1, // mainnet
  paymentCredential: new KeyHash.KeyHash({
    hash: new Uint8Array(28)
  })
});

// Convert to Bech32 string
const bech32 = Address.toBech32(address);
console.log(bech32); // "addr1..."
```

## Parsing Existing Addresses

Parse a Bech32 address string into an `EnterpriseAddress` instance:

```typescript twoslash
import { Address, EnterpriseAddress } from "@evolution-sdk/evolution";

const bech32 = "addr1vx2kd28nq8ac5prwg32hhvudlwggpgfp8utlyqxu6wqgz6cevnrgl";

const address = Address.fromBech32(bech32) as EnterpriseAddress.EnterpriseAddress;

console.log("Network ID:", address.networkId);
console.log("Payment:", address.paymentCredential);
```

## Parsing Enterprise Addresses

Parse and identify enterprise addresses:

```typescript twoslash
import { Address, EnterpriseAddress } from "@evolution-sdk/evolution";

const bech32Address = "addr1vx2kd28nq8ac5prwg32hhvudlwggpgfp8utlyqxu6wqgz62f79qsdmm5dsknt9ecr5w468r9ey0fxwkdrwh08ly3tu9s5wdfk2";

// Parse Bech32 to EnterpriseAddress
const address = Address.fromBech32(bech32Address) as EnterpriseAddress.EnterpriseAddress;

console.log("This is an enterprise address");
console.log("Payment credential:", address.paymentCredential);
```

## Converting from Bech32

Parse both key-based and script-based enterprise addresses:

```typescript twoslash
import { EnterpriseAddress, ScriptHash, Address } from "@evolution-sdk/evolution";

// Script address example
const scriptAddr = new EnterpriseAddress.EnterpriseAddress({
  networkId: 1, // mainnet
  paymentCredential: new ScriptHash.ScriptHash({
    hash: new Uint8Array(28) // 28-byte script hash
  })
});

// Convert to Bech32 string
const bech32 = Address.toBech32(scriptAddr);
console.log("Script enterprise address:", bech32);
```

## Format Details

**Bech32 Prefix**: `addr` (mainnet) or `addr_test` (testnet)  
**Length**: 29 bytes raw / ~59 characters Bech32  
**Header Bits**: `0110xxxx` (enterprise address type)  
**Size Advantage**: Half the size of base addresses (29 vs 57 bytes)

## Comparison with Base Addresses

| Feature | Enterprise | Base |
|---------|-----------|------|
| Payment credential | ✅ Yes | ✅ Yes |
| Staking credential | ❌ No | ✅ Yes |
| Can receive funds | ✅ Yes | ✅ Yes |
| Can delegate stake | ❌ No | ✅ Yes |
| Earns staking rewards | ❌ No | ✅ Yes |
| Size | 29 bytes | 57 bytes |
| Use case | Exchanges, scripts | User wallets |

## Advantages

**Smaller Size**: Half the bytes of base addresses. Lower transaction fees when used as inputs/outputs.

**Simplicity**: No stake key management. One key to secure instead of two.

**Performance**: Faster to generate and process. Good for high-volume applications.

**Privacy**: Each address can have independent payment credential. No stake key linkability.

## Limitations

**No Staking Rewards**: Funds at enterprise addresses don't participate in staking. Lost opportunity for passive rewards.

**No Delegation**: Cannot contribute to decentralization via stake delegation.

**User Expectation**: Most users expect base addresses with staking enabled by default.

## Best Practices

**Use Case Fit**: Only use enterprise addresses when you specifically don't want staking. Base addresses are the default for good reason.

**Exchange Operations**: Ideal for custodial platforms managing many user deposits. Simplifies key management at scale.

**Script Addresses**: Natural choice for smart contracts that don't need staking capability.

**Documentation**: Clearly communicate to users if you're using enterprise addresses. They might expect staking rewards.

## Security Considerations

**Single Key**: Only one key to secure (payment). Simpler security model, but no separation of spending vs staking authority.

**Key Rotation**: Changing payment key requires generating new address and moving funds. No independent stake key rotation.

**Custodial Risk**: Common in exchanges. Users trust platform with keys. Non-custodial alternatives preferred when possible.
