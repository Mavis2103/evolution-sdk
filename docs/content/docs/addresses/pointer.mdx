# Pointer Addresses

Pointer addresses contain a payment credential plus a pointer to an on-chain stake registration certificate. They reference stake information indirectly via blockchain coordinates instead of embedding a stake credential.

## Structure

```
Pointer Address = Payment Credential + Pointer (slot, txIndex, certIndex)
```

**Payment Credential**: Controls who can spend UTXOs  
**Pointer**: References stake registration cert location on-chain

## When to Use

- **Rare**: Pointer addresses are uncommon in practice
- **Legacy support**: Understanding historical transactions
- **Advanced scenarios**: Specific protocol-level optimizations

**Note**: Base addresses are preferred in almost all cases. Pointer addresses add complexity with minimal benefit.

## Manual Construction (Not Recommended)

This example shows the structure but is **not recommended for production**:

```typescript
import { PointerAddress, Pointer } from "@evolution-sdk/evolution";

const pointerAddr = PointerAddress.make({
  _tag: "PointerAddress" as const,
  networkId: 0,
  paymentCredential: {
    _tag: "KeyHash" as const,
    hash: "0".repeat(56)
  },
  pointer: Pointer.Pointer.make({
    slot: 0n,
    txIndex: 0n,
    certIndex: 0n
  })
});

console.log("Pointer address (deprecated):", pointerAddr);
```

## Pointer Components

The pointer identifies where the stake registration certificate lives on-chain:

- **slot**: Block slot number containing the registration
- **txIndex**: Transaction position within that block  
- **certIndex**: Certificate position within that transaction

This triplet uniquely identifies the stake registration certificate.

## Format Details

**Bech32 Prefix**: `addr` (mainnet) or `addr_test` (testnet)  
**Length**: Variable (34-48 bytes depending on pointer values)  
**Header Bits**: `0100xxxx` (pointer address type)

## Comparison with Base Addresses

| Feature | Pointer | Base |
|---------|---------|------|
| Payment credential | ✅ Yes | ✅ Yes |
| Stake credential | Via pointer | Direct embed |
| Size | Variable | Fixed 57 bytes |
| Complexity | High | Low |
| Adoption | Rare | Universal |
| Recommended | ❌ No | ✅ Yes |

## Why Pointer Addresses Exist

Pointer addresses were designed to reduce address size by referencing stake info instead of embedding it. In practice:

**Original Goal**: Save 28 bytes per address by using pointer instead of full stake credential.

**Reality**: Minimal savings, added complexity. On-chain stake registrations must be permanent and indexed. Wallet software needs additional logic to resolve pointers.

**Outcome**: Ecosystem converged on base addresses. Simpler, more predictable, better wallet support.

## Stake Registration Requirement

For a pointer address to work:

1. A stake registration certificate must exist on-chain at the pointer location
2. That certificate must register a stake credential
3. The pointer (slot, txIndex, certIndex) must be valid and resolvable

If any of these fail, the address is invalid.

## Best Practices

**Don't Use Them**: Unless you have a very specific reason, use base addresses instead. They're simpler, more widely supported, and the ecosystem standard.

**Legacy Support**: If your application encounters pointer addresses, handle them gracefully for backwards compatibility.

**Documentation**: If you must use pointer addresses, document why and ensure users understand the trade-offs.

## Security Considerations

**Pointer Resolution**: The on-chain certificate referenced by the pointer must remain accessible. If the chain reorganizes or the cert becomes unreachable, the address could become problematic.

**Complexity**: More complexity = more potential for bugs. Pointer addresses add resolution logic that can fail.

**Tooling**: Limited wallet and tooling support. Users may have trouble with pointer addresses in popular wallets.

## Historical Context

Pointer addresses were part of the original Shelley address design. The idea was to optimize for size and flexibility. However, practical considerations led to base addresses becoming the standard:

- **Simplicity**: Base addresses are self-contained
- **Predictability**: No pointer resolution needed
- **Compatibility**: Universal wallet support
- **Performance**: No lookups required
