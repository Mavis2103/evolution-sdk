# Reward Addresses

Reward addresses (also called stake addresses) contain only a staking credential. They're used to receive staking rewards and control stake delegation, but cannot hold regular UTXOs.

## Structure

```
Reward Address = Staking Credential Only
```

**Staking Credential**: Controls delegation and reward withdrawal  
**No Payment**: Cannot receive or spend regular UTXOs

## When to Use

- **Reward Withdrawal**: Claiming accumulated staking rewards
- **Delegation Management**: Delegating stake without exposing payment keys
- **Staking Pools**: Pool registration and reward distribution
- **Governance**: Voting rights based on stake

## Automatic Creation

Reward addresses are automatically associated with base addresses:

```typescript
import { createClient } from "@evolution-sdk/evolution";

const client = createClient({
  network: "preprod",
  wallet: {
    type: "seed",
    mnemonic: "your twenty-four word seed phrase here...",
    accountIndex: 0
  }
});

// Base address (payment + stake)
const paymentAddress = await client.address();
console.log("Payment address:", paymentAddress);
// addr_test1qz...

// Reward address (stake only) - derived from same stake key
// Used for withdrawing rewards from the base address above
```

## Understanding Reward Addresses

Reward addresses are not constructed directly in the SDK. They are Bech32-encoded addresses with the `stake`/`stake_test` prefix that can be parsed using the Core `AddressStructure` module.

However, reward addresses are a special case - they use a different address format called `RewardAccount` which is not yet fully exposed in the current Core module. The SDK handles reward addresses internally through wallet operations.

## Relationship with Base Addresses

Base and reward addresses share the same stake credential:

```
Base Address:    addr1qx... (payment + stake)
                           ↓ (same stake key)
Reward Address:  stake1uy... (stake only)
```

When you delegate or withdraw rewards, you use the reward address. The rewards accumulate at the reward address, not the payment address.

## Delegation Pattern

Delegate stake using reward address:

```typescript
import { createClient } from "@evolution-sdk/evolution";

const client = createClient({
  network: "preprod",
  wallet: {
    type: "seed",
    mnemonic: "your seed phrase...",
    accountIndex: 0
  }
});

// To delegate stake or withdraw rewards, you need the reward address
// The SDK handles this automatically through the wallet's stake key
```

## Reward Withdrawal

Withdraw accumulated rewards:

```typescript
import { createClient } from "@evolution-sdk/evolution";

const client = createClient({
  network: "preprod",
  wallet: {
    type: "seed",
    mnemonic: "your seed phrase...",
    accountIndex: 0
  },
  provider: {
    type: "blockfrost",
    projectId: "your-project-id"
  }
});

// Build transaction to withdraw rewards
async function withdrawRewards() {
  const tx = await client
    .newTx()
    .withdrawRewards() // Withdraws from reward address to payment address
    .build();
  
  const signed = await client.signTx(tx);
  const txHash = await client.submitTx(signed);
  
  return txHash;
}
```

## Format Details

**Bech32 Prefix**: `stake` (mainnet) or `stake_test` (testnet)  
**Length**: 29 bytes raw / ~59 characters Bech32  
**Header Bits**: `1110xxxx` (reward address type)

## Key Separation

Reward addresses enable separation of concerns:

| Capability | Payment Key | Stake Key |
|------------|-------------|-----------|
| Spend UTXOs | ✅ Yes | ❌ No |
| Receive funds | ✅ Yes | ❌ No |
| Delegate stake | ❌ No | ✅ Yes |
| Withdraw rewards | ❌ No | ✅ Yes |
| Vote (governance) | ❌ No | ✅ Yes |

This separation allows you to:
- Keep payment key in cold storage
- Use stake key for delegation (hot wallet)
- Rotate keys independently

## Best Practices

**Automatic Usage**: In most cases, you don't manually construct reward addresses. The wallet and SDK handle them automatically.

**Delegation**: Use reward addresses for delegation transactions. The payment address is not directly involved.

**Reward Claiming**: Rewards must be explicitly withdrawn. They don't automatically appear at your payment address.

**Key Security**: Stake key should be secured, but doesn't need the same cold storage treatment as payment key (since it can't spend funds).

## Security Considerations

**Limited Risk**: Compromised stake key can change delegation or withdraw rewards, but cannot spend funds from payment address.

**Delegation Changes**: Attacker with stake key could change your delegation, but your funds remain safe.

**Reward Theft**: Compromised stake key allows withdrawal of rewards, but not spending of principal funds.

**Recovery**: If stake key is compromised, create new base address with different stake key and transfer funds.

### Compound Rewards

Automatically re-delegate withdrawn rewards:

```typescript
import { createClient } from "@evolution-sdk/evolution";

const client = createClient({
  network: "preprod",
  wallet: {
    type: "seed",
    mnemonic: "your seed phrase...",
    accountIndex: 0
  },
  provider: {
    type: "blockfrost",
    projectId: "your-project-id"
  }
});

async function compoundRewards() {
  // Withdraw rewards to payment address
  const tx = await client
    .newTx()
    .withdrawRewards() // Adds withdrawal to transaction
    .build();
  
  // Rewards now available at payment address, automatically re-staked
  const signed = await client.signTx(tx);
  return await client.submitTx(signed);
}
```
